# Version 1.1 ERP Upload Implementation Plan

## Status

Planning only. No application code is implemented or authorized by this document.

## Purpose

Define a practical, auditable implementation path for importing SAP or Oracle procurement exports into AI Procurement Copilot through one multi-sheet Excel workbook.

Direct SAP or Oracle API integration is explicitly outside the MVP.

## Design Review Conclusion

The existing ERP planning set is directionally strong and internally consistent:

- `ERP_UPLOAD_DATA_MODEL.md` correctly separates ERP-specific fields from canonical Copilot fields.
- `SAP_ORACLE_FIELD_MAPPING.md` provides a useful starting map while acknowledging configuration differences.
- `ERP_UPLOAD_TEMPLATE_SPEC.md` defines a workable low-code workbook standard.
- `ERP_DATA_GAP_ANALYSIS.md` correctly separates ERP, manual, external, and derived data.

The main implementation risk is not reading Excel. The real risk is incorrectly matching suppliers, materials, currencies, units, periods, and business meanings. Therefore, Version 1.1 must prioritize validation, auditability, and safe rejection over convenience.

## Recommendation

Build a file-based ERP ingestion MVP first.

The MVP should:

- Accept one `.xlsx` workbook.
- Support SAP and Oracle mapping profiles.
- Normalize data into canonical tables.
- Show mapping and validation results before analysis.
- Allow safe manual correction of ambiguous mappings.
- Preserve raw source data and transformation metadata.
- Feed only validated records into existing modules.

Do not build live ERP connectivity, write-back, automatic supplier-master changes, or autonomous mapping in the MVP.

# 1. Phased MVP Scope

## Phase 0 — Design Lock and Sample Data

### Objective

Freeze the first implementation boundary and obtain representative test extracts.

### Deliverables

- One synthetic SAP workbook
- One synthetic Oracle workbook
- One deliberately poor-quality workbook
- Canonical column dictionary
- Mapping profile format
- Validation severity matrix
- Data lineage standard

### Exit criteria

- All seven priority sheets have sample data.
- Required fields are agreed.
- Blocking versus non-blocking validation rules are agreed.
- No code is written before these decisions are approved.

### Estimated effort

4–6 hours

## Phase 1 — Workbook Intake and Structural Validation

### Included sheets

1. `Supplier_Master`
2. `RFQ_Quotes`
3. `Purchase_Orders`
4. `Receipts`
5. `Supplier_Performance`
6. `Material_Master`
7. `Cost_Drivers`

### Objective

Read the workbook safely and determine whether it is structurally usable.

### Capabilities

- File type and size validation
- Required sheet detection
- Header extraction
- Duplicate-header detection
- Empty-sheet detection
- Unsupported-sheet logging
- Row-count summary
- Workbook metadata capture
- Read-only preview

### Exit criteria

- Invalid files fail clearly.
- All accepted sheets display row and column summaries.
- No business analysis runs before structural validation passes.

### Estimated effort

6–8 hours

## Phase 2 — Column Mapping and Profile Management

### Objective

Map SAP or Oracle source columns to canonical Copilot columns.

### Capabilities

- Select ERP profile: SAP, Oracle, Custom
- Suggested column mapping
- Manual mapping overrides
- Required-field warnings
- Transformation preview
- Save mapping profile as JSON
- Load an existing mapping profile
- Version and approve mapping profiles

### Guardrails

- No silent auto-acceptance for ambiguous required fields.
- Supplier and material identifiers cannot be mapped from descriptions without confirmation.
- Currency and unit columns require explicit mapping.
- Mapping changes are logged.

### Exit criteria

- Each required canonical field is mapped or explicitly marked missing.
- The user can review every source-to-target mapping.
- Mapping profiles are reusable and auditable.

### Estimated effort

8–12 hours

## Phase 3 — Canonical Normalization and Matching

### Objective

Convert accepted source data into canonical tables.

### Capabilities

- Supplier ID normalization
- Material ID normalization
- Date parsing
- Currency preservation and normalization
- Unit preservation and normalization
- Duplicate detection
- Referential-integrity validation
- Source-record lineage
- Data-confidence tagging

### Supplier matching hierarchy

1. Exact canonical supplier ID
2. Exact ERP supplier number
3. Tax ID or legal identifier
4. Legal name plus country
5. Fuzzy name suggestion requiring human approval

### Material matching hierarchy

1. Exact canonical material ID
2. Exact ERP material or item number
3. Manufacturer part number
4. Specification plus unit plus category
5. Description suggestion requiring human approval

### Exit criteria

- No ambiguous supplier or material match is silently accepted.
- Original and normalized currency and units remain available.
- Conflicting duplicates are quarantined.
- Every accepted record retains source lineage.

### Estimated effort

12–18 hours

## Phase 4 — Validation Gate and Upload UX

### Objective

Make data quality understandable to a procurement user before analysis begins.

### Capabilities

- Overall upload status
- Accepted, warned, quarantined, and rejected row counts
- Missing-field summary
- Supplier-match summary
- Material-match summary
- Currency and unit issues
- Date coverage
- Historical-period coverage
- Module eligibility matrix
- Downloadable validation report

### Status model

- PASS
- PASS WITH WARNINGS
- HUMAN REVIEW REQUIRED
- BLOCKED

### Exit criteria

- A user can understand why a workbook is accepted or blocked.
- Blocking problems prevent analysis.
- Non-blocking problems lower confidence and remain visible downstream.

### Estimated effort

8–12 hours

## Phase 5 — Existing Module Integration

### Objective

Feed validated canonical tables into existing Copilot modules without weakening Version 1.0 governance.

### Initial integrations

- RFQ comparison
- TCO
- Supplier performance
- Supplier 360
- SRM classification
- Allocation and negotiation
- Should-cost inputs

### Integration rule

Existing modules must consume canonical fields only. They must not depend directly on SAP or Oracle source labels.

### Exit criteria

- Existing demo and single-sheet workflows remain functional.
- ERP-upload records produce the same governed outputs as equivalent canonical demo data.
- Missing data produces provisional or blocked outcomes rather than fabricated values.

### Estimated effort

10–16 hours

## Phase 6 — Historical Procurement Analytics Foundation

### Objective

Use dated ERP records to support reporting-period selection and trend-ready data.

### Included capabilities

- All Available Data
- Custom Date Range
- Earliest and latest record dates
- Undated-record count
- Current versus comparison period metadata
- Snapshot versus transaction classification
- Partial-period warnings

### Deferred from MVP

- Full YoY dashboards across every module
- Rolling 12-month forecasting
- advanced seasonality
- live market-index integration

### Exit criteria

- Each enabled dataset declares a governing date.
- Date filtering is consistent.
- Undated records are excluded from trends and disclosed.
- Incomplete-period comparisons are qualified.

### Estimated effort

8–12 hours

## Phase 7 — Hardening and Release Validation

### Objective

Prove the upload workflow is safe, repeatable, and regression-resistant.

### Capabilities

- Performance tests
- Large-file tests
- Corrupt workbook tests
- adversarial mapping tests
- cross-sheet consistency tests
- export-integrity tests
- mobile UX testing
- user acceptance testing

### Exit criteria

- No open Critical defect
- No open unmitigated Major defect
- Stable v1.0 workflows remain intact
- SAP and Oracle synthetic workbooks pass
- poor-quality workbook fails safely

### Estimated effort

10–16 hours

# 2. Recommended MVP Boundary

## In Scope

- One `.xlsx` upload
- Seven priority sheets
- SAP, Oracle, and Custom mapping profiles
- Manual mapping confirmation
- Supplier and material matching
- Currency and unit normalization
- Date normalization
- Duplicate and missing-data governance
- Validation report
- Canonical-table generation
- Integration with core procurement modules
- Basic historical coverage and date filtering

## Out of Scope

- Direct SAP API
- Direct Oracle API
- ERP write-back
- automatic master-data updates
- real-time synchronization
- user authentication and enterprise SSO
- large-scale data warehouse architecture
- background jobs
- external commodity-data APIs
- contract text extraction
- autonomous fuzzy matching
- production security certification

## Why this boundary is correct

It maximizes interview and business value while keeping complexity controlled. It demonstrates enterprise data thinking without pretending to be a production ERP integration product.

# 4. Streamlit Upload and Mapping UX

## Recommended Flow

### Step 1 — Select Upload Mode

Options:

- Demo Data
- Existing RFQ File
- ERP Procurement Workbook

### Step 2 — Upload Workbook

Display:

- file name
- size
- detected sheets
- extraction date
- workbook status

### Step 3 — Select ERP Profile

Options:

- SAP ECC
- SAP S/4HANA
- Oracle EBS
- Oracle Fusion
- Custom

Do not claim automatic ERP detection as certain. It may be offered as a suggestion only.

### Step 4 — Review Sheets

For each sheet display:

- detected status
- row count
- required-column coverage
- warnings
- preview

### Step 5 — Map Columns

Use an editable table with:

- Source column
- Suggested canonical field
- User-selected canonical field
- Transformation
- Required status
- Confidence
- Validation message

### Step 6 — Resolve Entity Matches

Display ambiguous supplier and material matches in a review queue.

Actions:

- Accept suggested match
- Select another existing entity
- Create temporary canonical entity
- Reject row

### Step 7 — Review Validation

Dashboard sections:

- Workbook readiness
- Missing critical fields
- Supplier match rate
- Material match rate
- Currency issues
- Unit issues
- Date coverage
- duplicates
- module eligibility

### Step 8 — Confirm Processing

The user must explicitly approve processing after warnings are reviewed.

### Step 9 — Analysis

Show a persistent banner:

- ERP profile
- reporting period
- records accepted
- records excluded
- data confidence
- validation status

## Low-Code Principle

The user should be able to complete mapping through dropdowns and tables. No Python, SQL, JSON editing, or schema knowledge should be required.

# 7. Risks and Mitigations

## Risk: ERP fields differ from the documented mapping

Mitigation:

- Mapping profiles
- manual mapping
- source-column retention
- real extract validation before coding is declared complete

## Risk: Incorrect supplier merge

Mitigation:

- exact-ID priority
- no silent fuzzy merge
- human approval queue
- lineage report

## Risk: Unit or currency corruption

Mitigation:

- preserve originals
- block incompatible comparison
- require explicit FX and conversion metadata

## Risk: False confidence from incomplete history

Mitigation:

- coverage metadata
- snapshot versus transaction flag
- undated exclusion counts
- incomplete-period warnings

## Risk: Version 1.0 regression

Mitigation:

- develop only on `v1.1-development`
- preserve current loaders
- add ERP mode as a separate path
- regression-test existing flows

## Risk: Large workbook performance

Mitigation:

- row limits in MVP
- preview sampling
- cached reads
- staged processing
- clear file-size limits

## Risk: Spreadsheet security

Mitigation:

- do not execute formulas or macros
- accept `.xlsx` only
- reject macro-enabled files
- sanitize exported text
- limit file size

## Risk: Overbuilding before real data is available

Mitigation:

- require one SAP and one Oracle sample
- implement seven priority sheets first
- delay invoices, contracts, and quality if schedule requires

# 8. Rollback Strategy

## Branching

- Keep `main` and `release/v1.0.0` untouched.
- Implement only on `v1.1-development` or feature branches created from it.

## Feature isolation

- Add ERP upload as a separate data-source option.
- Do not replace the current RFQ loader initially.
- Use feature flags for unfinished modules.

## Data isolation

- Process uploads in memory for the MVP.
- Do not write uploaded data to external systems.
- Do not persist sensitive source data by default.

## Release rollback

If the ERP workflow is unstable:

- disable the ERP upload option
- retain all Version 1.0 workflows
- revert the ERP integration commit set
- preserve mapping and validation documentation

## Trigger conditions for rollback

- Existing upload workflow breaks
- recommendation governance is bypassed
- supplier matching accepts ambiguous records silently
- currency or unit normalization produces incorrect comparisons
- uploaded data persists unexpectedly
- export audit lineage is lost

# 9. Acceptance Criteria

Version 1.1 ERP Upload MVP is accepted only when:

## File and Structure

- `.xlsx` upload works.
- Required sheets are detected.
- Missing or corrupt structures fail clearly.

## Mapping

- SAP, Oracle, and Custom profiles work.
- Every required field is mapped or declared missing.
- Mapping changes are auditable.

## Matching

- Supplier and material exact matches work.
- Ambiguous matches require human review.
- No silent fuzzy merge occurs.

## Governance

- Original and normalized currencies are retained.
- Original and normalized units are retained.
- Missing critical data blocks affected modules.
- Non-critical gaps reduce confidence.
- Duplicates and conflicts are visible.

## Historical Data

- Governing dates are defined.
- Date coverage is displayed.
- Undated records are disclosed.
- incomplete periods are labelled.

## Integration

- RFQ, TCO, Supplier 360, SRM, should-cost, and negotiation modules consume canonical tables.
- Existing Version 1.0 workflows continue to pass.

## Auditability

- Mapping report is downloadable.
- Validation report is downloadable.
- Rejected and quarantined rows are exportable.
- Source lineage is preserved.

## User Experience

- A non-technical procurement user can complete the workflow without coding.
- Errors are written in business language.
- The user can see which modules are eligible to run.

## Release Quality

- All unit, integration, regression, and adversarial tests pass.
- No Critical defect remains open.
- No unmitigated Major defect remains open.
- Human procurement review confirms the results are decision-useful.
